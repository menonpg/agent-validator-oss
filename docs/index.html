What is the Agent Validator?

A governance pipeline that validates AI agents against 28 rules before they're deployed into your production environment.

Every AI agent that touches sensitive data, internal APIs, or critical workflows must pass validation before deployment. The system clones the agent's GitHub repository, runs 5 categories of checks (secret scanning, dependency audits, AST analysis, LLM-judged capability scoring, and live runtime tests), and produces a tiered governance scorecard.

New rules are added by editing markdown files in Git — no code changes required. The validator's own memory (what patterns pass and fail over time) is stored in GCS and updated automatically after each run, using the same soul.py pattern as the SoulMate platform.

The 5 Check Types

🔍 regex_scan

Greps source files for hardcoded secrets (API keys, passwords, tokens), banned import patterns, and forbidden outbound domains. Fast — runs in milliseconds. Used for SEC-001, SEC-001, SEC-002.

📦 dep_scan

Parses requirements.txt and runs pip-audit against the OSV vulnerability database. Flags unpinned versions (e.g. fastapi without a version) and known CVEs. Used for SEC-002, SEC-003.

🌳 ast_check

Parses Python source into an AST and checks for structural requirements: /health endpoint, retry decorators, MAX_STEPS constant, structlog import, OTEL traces, parameterized BigQuery queries. Deterministic — no false positives from comments.

🤖 llm_judge

Uses Gemini 2.0 Flash with a soul.py SOUL.md identity ("strict but fair governance auditor") to score capability fidelity, scope adherence, jailbreak resistance, and PII leakage protection. Score ≥7/10 to pass quality gates.

🚀 runtime_test

Spins up the agent locally in a subprocess, hits its /health endpoint, and verifies the response schema. Catches agents that crash on startup, have missing dependencies, or return malformed health responses.

The 3 Verdict Tiers

🔴

Hard Gates (10 rules)

Any failure = REJECTED. Covers critical security and structure: no hardcoded secrets, no CVEs, required files present, test coverage, BigQuery parameterization, PII redaction in logs.

🟡

Soft Gates (16 rules)

Failures produce warnings and reduce the quality score. Deployment allowed. Covers governance, observability, reliability, alignment: pinned model versions, OTEL traces, retry logic, AI disclosure, jailbreak resistance.

🔵

Quality Score (0–100)

LLM judge scores capability fidelity and scope adherence. Score reflects how well the agent implements what it claims. Tracked in MEMORY.md over time — institutional learning across all runs.

soul.py Integration

The validator uses soul.py in three ways:

1. Rules-as-Policy Memory — rules/*.md files ARE the validator's long-term memory of "what good looks like." Human-editable, git-versioned, PR-reviewed. Adding a new rule = editing markdown only.

2. Judge Identity — engine/soul/SOUL.md defines the Gemini judge's persona: "I am a strict but fair AI governance auditor. PII leakage score ≤4 regardless of other factors." The judge's character is persistent and auditable.

3. Institutional Memory — engine/soul/MEMORY.md in GCS is auto-updated after each run with patterns observed (which teams pass, which rules fail most, model version trends). The validator learns over time.
